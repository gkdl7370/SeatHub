# 4일차 작업 기록

## 작업 목표

1~3일차에는 프로젝트 문서, 로컬 실행 환경, 공통 응답과 예외 처리 구조를 만들었습니다.

4일차에는 실제 도메인 구현으로 넘어가기 전에 다음 두 가지를 먼저 정리했습니다.

- DB 스키마를 코드와 함께 관리할 수 있는 기반 마련
- 첫 번째 도메인인 회원 가입 흐름 구현

## Flyway 기반 DB 마이그레이션 추가

`local` 프로파일은 JPA `ddl-auto: validate`를 사용합니다.

이 설정은 애플리케이션 실행 시 Entity와 실제 DB 테이블이 맞는지 검증합니다. 대신 테이블을 자동으로 만들어주지는 않습니다.

그래서 운영에 가까운 방식으로 스키마를 관리하기 위해 Flyway를 추가했습니다.

추가한 마이그레이션:

- `users`
- `user_roles`

## 회원 도메인 추가

추가한 주요 클래스:

- `User`
- `UserRole`
- `UserRepository`

회원은 기본적으로 `USER` 권한을 갖습니다.

비밀번호는 원문 저장을 피하기 위해 `PasswordEncoder`를 통해 암호화된 값만 저장합니다.

## 회원가입 API 추가

추가한 API:

```text
POST /api/v1/auth/signup
```

요청값:

```json
{
  "email": "user@example.com",
  "password": "password123",
  "name": "테스트회원"
}
```

처리 흐름:

```text
요청 검증
-> 이메일 중복 확인
-> 비밀번호 암호화
-> 회원 저장
-> 가입 결과 응답
```

## 예외 처리

중복 이메일에 대한 에러 코드를 추가했습니다.

```text
DUPLICATE_EMAIL
```

이미 가입된 이메일로 회원가입을 요청하면 `409 Conflict` 응답을 반환합니다.

## 테스트

추가한 테스트:

- 회원가입 시 회원이 저장되는지 확인
- 비밀번호가 원문이 아닌 암호화된 값으로 저장되는지 확인
- 중복 이메일 가입 요청 시 예외가 발생하는지 확인

## 왜 이렇게 했는가

회원가입은 이후 로그인, 예약, 결제, 관리자 권한의 출발점입니다.

처음부터 JWT까지 한 번에 구현하면 범위가 커지기 때문에, 오늘은 다음 작업으로 이어질 수 있는 최소 단위인 회원 생성과 중복 검증까지만 구현했습니다.
