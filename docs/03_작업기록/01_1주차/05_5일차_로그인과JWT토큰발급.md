# 5일차 작업 기록

## 작업 목표

4일차에는 회원가입 API와 회원 테이블 마이그레이션을 만들었습니다.

5일차에는 회원가입 다음 흐름인 로그인 API를 추가했습니다.

이번 작업의 목표는 다음과 같습니다.

- 이메일과 비밀번호로 로그인할 수 있는 구조 만들기
- 비밀번호 원문 비교가 아니라 `PasswordEncoder.matches`로 검증하기
- 로그인 성공 시 Access Token 응답 구조 만들기
- 로그인 실패 시 공통 예외 응답으로 처리하기

## 추가한 기능

### 로그인 API

```text
POST /api/v1/auth/login
```

요청값:

```json
{
  "email": "user@example.com",
  "password": "password123"
}
```

응답값:

```json
{
  "success": true,
  "data": {
    "tokenType": "Bearer",
    "accessToken": "header.payload.signature",
    "expiresIn": 3600
  },
  "traceId": "request-trace-id"
}
```

## 인증 처리 흐름

```text
이메일로 회원 조회
-> 회원이 없으면 INVALID_LOGIN 예외
-> 비밀번호 검증
-> 비밀번호가 다르면 INVALID_LOGIN 예외
-> Access Token 생성
-> 로그인 응답 반환
```

## JWT 토큰 발급 구조

이번 단계에서는 외부 JWT 라이브러리를 추가하지 않고, Java 표준 API 기반으로 간단한 HS256 토큰 생성 구조를 만들었습니다.

토큰에 포함한 값:

- `sub`: 회원 ID
- `email`: 회원 이메일
- `iat`: 발급 시각
- `exp`: 만료 시각

아직 인증 필터에서 토큰을 검증하는 단계까지는 구현하지 않았습니다.

현재 단계의 목적은 로그인 성공 시 토큰을 발급하는 흐름을 먼저 분리하는 것입니다.

## 예외 처리

로그인 실패 시 이메일이 틀렸는지, 비밀번호가 틀렸는지 구분하지 않고 같은 에러를 반환합니다.

```text
INVALID_LOGIN
```

이렇게 하는 이유는 로그인 실패 응답만 보고 가입된 이메일인지 추측하기 어렵게 하기 위해서입니다.

## 테스트

추가한 테스트:

- 로그인 성공 시 Bearer 토큰이 반환되는지 확인
- 존재하지 않는 이메일로 로그인하면 실패하는지 확인
- 비밀번호가 틀리면 로그인에 실패하는지 확인

## 왜 이렇게 했는가

로그인은 이후 예약, 결제, 관리자 API에서 사용자 식별의 기반이 됩니다.

다만 오늘은 인증 필터와 SecurityContext 연동까지 한 번에 구현하지 않고, 로그인 성공과 토큰 발급 책임을 먼저 분리했습니다.

이렇게 하면 다음 단계에서 토큰 검증 필터를 추가할 때 기존 로그인 흐름을 크게 바꾸지 않고 이어갈 수 있습니다.
