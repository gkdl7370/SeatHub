# 6일차 작업 기록

## 작업 목표

5일차에는 로그인 성공 시 Access Token을 발급하는 흐름까지 만들었습니다.

6일차에는 발급된 토큰을 실제 요청에서 사용할 수 있도록 인증 흐름을 연결했습니다.

이번 작업의 목표는 다음과 같습니다.

- `Authorization: Bearer {token}` 형식의 요청 헤더 읽기
- JWT 서명과 만료 시간 검증
- 토큰의 회원 ID를 기준으로 회원 정보 조회
- Spring Security의 인증 객체로 현재 사용자 등록
- 현재 로그인한 회원 정보를 조회하는 API 추가

## 추가한 기능

### JWT 검증

`JwtTokenProvider`에 토큰 검증 기능을 추가했습니다.

검증 기준은 다음과 같습니다.

- 토큰이 `header.payload.signature` 형식인지 확인
- 서버의 secret으로 다시 서명을 만들고 기존 서명과 비교
- payload의 `exp` 값이 현재 시각보다 이후인지 확인
- payload의 `sub` 값을 회원 ID로 사용

현재는 외부 JWT 라이브러리 없이 Java 표준 API와 Jackson을 사용했습니다.

## 인증 필터

`JwtAuthenticationFilter`를 추가했습니다.

처리 흐름은 다음과 같습니다.

```text
요청 헤더에서 Authorization 확인
-> Bearer 토큰 추출
-> JWT 서명과 만료 시간 검증
-> 토큰의 회원 ID로 회원 조회
-> AuthenticatedUser 생성
-> SecurityContext에 인증 정보 저장
```

이 구조가 들어가면서 로그인 이후의 API에서는 매번 이메일과 비밀번호를 다시 받지 않고,
토큰만으로 현재 사용자를 식별할 수 있게 되었습니다.

## 내 정보 조회 API

```text
GET /api/v1/auth/me
```

요청 예시:

```http
GET /api/v1/auth/me
Authorization: Bearer {accessToken}
```

응답 예시:

```json
{
  "success": true,
  "data": {
    "id": 1,
    "email": "user@example.com",
    "name": "테스트회원",
    "roles": ["USER"]
  },
  "traceId": "request-trace-id"
}
```

## 왜 이렇게 했는가

예약, 결제, 취소, 내 예약 조회 같은 기능은 모두 “요청한 사용자가 누구인가”를 알아야 합니다.

그래서 회원가입과 로그인 다음 단계로 JWT 인증 필터를 먼저 연결했습니다.

이제 이후 기능에서는 컨트롤러나 서비스에서 회원 ID를 직접 요청값으로 받지 않고,
인증된 사용자 정보를 기준으로 처리할 수 있습니다.

예를 들어 예약 생성 API에서는 다음과 같은 방향으로 구현할 수 있습니다.

```text
요청 사용자 확인
-> 예약할 회차와 좌석 확인
-> 현재 사용자 기준으로 예약 생성
-> 결제 대기 상태로 변경
```

이렇게 하면 다른 사용자의 회원 ID를 임의로 넣어서 예약을 생성하는 문제를 막을 수 있습니다.

## 테스트

추가한 테스트는 다음과 같습니다.

- 생성한 JWT를 검증하고 회원 ID를 다시 꺼낼 수 있는지 확인
- 잘못된 형식의 토큰은 유효하지 않은 토큰으로 처리되는지 확인
- 회원가입, 로그인, 내 정보 조회 API가 연결되어 동작하는지 확인

실행 결과:

```text
gradle test
BUILD SUCCESSFUL
```
