# SeatHub 요구사항

## 목표

예약·결제 도메인에서 자주 발생하는 동시성, 상태 전이, 결제 대기 만료, 검색 성능, 운영 추적 문제를 백엔드 관점에서 구현하고 검증합니다.

---

## 사용자 요구사항

1. 사용자는 이메일과 비밀번호로 회원가입할 수 있다.
2. 사용자는 로그인 후 JWT 토큰을 발급받을 수 있다.
3. 사용자는 예약 가능한 상품 목록을 조회할 수 있다.
4. 사용자는 상품 상세 정보를 조회할 수 있다.
5. 사용자는 특정 상품의 회차 목록을 조회할 수 있다.
6. 사용자는 특정 회차의 좌석 목록과 예약 가능 여부를 조회할 수 있다.
7. 사용자는 예약 가능한 좌석을 선택해 예약을 생성할 수 있다.
8. 사용자는 결제 대기 중인 예약을 취소할 수 있다.
9. 사용자는 본인의 예약 목록을 조회할 수 있다.
10. 사용자는 본인의 예약 상세 정보를 조회할 수 있다.

---

## 관리자 요구사항

11. 관리자는 예약 상품을 등록할 수 있다.
12. 관리자는 상품별 회차를 등록할 수 있다.
13. 관리자는 회차별 좌석 또는 재고를 등록할 수 있다.
14. 관리자는 예약 상태, 기간, 상품명, 예약자 조건으로 예약을 검색할 수 있다.
15. 관리자는 결제 상태, 기간, 예약자 조건으로 결제 내역을 검색할 수 있다.

---

## 결제 요구사항

16. 시스템은 예약 생성 후 예약 상태를 `PAYMENT_PENDING`으로 변경한다.
17. 시스템은 결제 승인 성공 시 예약 상태를 `CONFIRMED`로 변경한다.
18. 시스템은 결제 실패 시 예약 상태를 `PAYMENT_FAILED`로 변경하고 좌석 점유를 해제한다.
19. 시스템은 결제 대기 시간이 초과된 예약을 `EXPIRED`로 변경하고 좌석 점유를 해제한다.

---

## 품질 요구사항

20. 같은 좌석에 여러 사용자가 동시에 예약을 요청해도 성공 예약은 1건만 생성되어야 한다.
21. 예약 상태가 변경될 때마다 상태 변경 이력을 저장해야 한다.
22. 외부 결제 시스템 호출은 DB 트랜잭션을 오래 점유하지 않도록 분리해야 한다.
23. 관리자 예약 검색 API는 N+1 문제를 만들지 않아야 한다.
24. 관리자 예약 검색 API는 주요 검색 조건에 맞는 인덱스를 사용해야 한다.
25. 모든 API 예외 응답은 일관된 형식을 가져야 한다.
26. 주요 API는 통합 테스트로 검증해야 한다.
27. 동일 좌석 동시 예약은 동시성 테스트로 검증해야 한다.
28. 로컬 환경은 Docker Compose로 실행할 수 있어야 한다.
29. GitHub Actions에서 테스트가 자동 실행되어야 한다.
30. README만 보고 프로젝트 실행과 핵심 설계를 이해할 수 있어야 한다.

## 1차 구현 범위

- 회원가입 / 로그인
- 상품 / 회차 / 좌석 등록 및 조회
- 예약 생성 / 취소
- 결제 준비 / 승인 / 실패 Mock
- 결제 대기 예약 만료 배치
- 관리자 예약 검색
- 동시성 테스트
- 성능 개선 보고서

## 2차 개선 범위

- Redis 분산 락 적용
- 일별 정산
- Outbox Event
- Prometheus / Grafana
- AWS EC2 / RDS 배포
