# SeatHub 아키텍처 초안

## 목표

SeatHub는 예약·결제 도메인에서 다음 문제를 다룹니다.

- 동일 좌석 중복 예약 방지
- 예약과 결제 상태 전이 관리
- 외부 결제 시스템과의 의존성 분리
- 관리자 검색 성능 개선
- 운영 추적을 위한 상태 이력과 traceId

---

## 전체 구조

```text
Client
  -> Spring Boot API
      -> Auth
      -> Product / Schedule / Seat
      -> Reservation
      -> Payment
      -> Admin Search
      -> Batch
  -> MySQL
  -> Redis
```

---

## 트랜잭션 경계

### 예약 생성

```text
1. 좌석 조회
2. 좌석 비관적 락 획득
3. 좌석 예약 가능 여부 확인
4. 예약 생성
5. 좌석 임시 점유
6. 예약 상태 이력 저장
7. 트랜잭션 커밋
```

### 결제 승인

```text
1. 결제 승인 요청 수신
2. 결제 결과 검증
3. 결제 정보 저장
4. 예약 상태 CONFIRMED 변경
5. 예약 상태 이력 저장
6. 트랜잭션 커밋
```

### 결제 대기 만료

```text
1. PAYMENT_PENDING 상태 중 만료 시간이 지난 예약 조회
2. 예약 상태 EXPIRED 변경
3. 좌석 점유 해제
4. 예약 상태 이력 저장
5. 트랜잭션 커밋
```

---

## 동시성 제어

1차 구현에서는 DB 비관적 락을 사용합니다.

선택 이유:

- 예약 도메인은 중복 예약 방지가 성능보다 우선입니다.
- MySQL 트랜잭션과 함께 정합성을 보장하기 쉽습니다.
- 구현과 테스트가 명확해 면접에서 설명하기 좋습니다.

2차 개선에서는 Redis 분산 락과 비교합니다.

## 성능 개선 대상

관리자 예약 검색 API를 성능 개선 대상으로 둡니다.

개선 흐름:

1. 단순 JPA 조회
2. N+1 발생 확인
3. DTO Projection 적용
4. 복합 인덱스 추가
5. 실행계획 비교
6. k6 부하 테스트
7. 성능 개선 보고서 작성
