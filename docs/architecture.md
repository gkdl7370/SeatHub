# SeatHub 아키텍처 초안

## 목표

SeatHub는 예약·결제 도메인에서 다음 문제를 다룹니다.

- 동일 좌석 중복 예약 방지
- 예약과 결제 상태 전이 관리
- 외부 결제 시스템과의 의존성 분리
- 관리자 검색 성능 개선
- 운영 추적을 위한 상태 이력과 traceId

---

## 기존 프로젝트에서 가져온 설계 관점

### One-Core-Architecture

SeatHub에서는 예약, 결제, 상품, 회차, 좌석을 명확한 도메인 경계로 나눕니다.

외부 결제 시스템, 관리자 정책, 알림 이벤트처럼 변경 가능성이 큰 영역은 핵심 예약 도메인에 직접 섞지 않고 인터페이스나 별도 모듈로 분리합니다.

적용 위치:

- `reservation`: 예약 상태 전이와 좌석 점유 규칙
- `payment`: 외부 결제 연동 추상화
- `admin`: 운영자 검색과 관리 기능
- `infra`: 외부 시스템 연동 구현

### Data-Flux

SeatHub에서는 배치와 성능 측정 관점을 적용합니다.

적용 위치:

- 결제 대기 만료 배치
- 대량 테스트 데이터 생성
- 관리자 예약 검색 성능 개선
- 성능 개선 전후 리포트 작성

### MassFlux-Gateway

SeatHub에서는 고처리량 요청 상황에서 병목을 확인하는 관점을 적용합니다.

적용 위치:

- 동일 좌석 동시 예약 테스트
- 예약 생성 요청 부하 테스트
- 락 대기 시간과 DB 병목 분석
- 예약 생성과 결제 후속 처리 분리

### kafka-practice

SeatHub 1차 구현에서는 동기 흐름으로 예약·결제를 구현합니다. 이후 2차 개선에서 이벤트 기반 구조를 적용할 수 있게 확장 지점을 남깁니다.

후보 이벤트:

- `ReservationCreatedEvent`
- `PaymentConfirmedEvent`
- `PaymentFailedEvent`
- `ReservationExpiredEvent`

2차 개선 후보:

- Kafka 또는 SQS 기반 이벤트 발행
- Consumer 실패 재시도
- DLQ 분리
- 이벤트 멱등 처리

### SimpleIoT.Gateway

SeatHub에서는 운영 추적 관점을 참고합니다.

적용 위치:

- traceId 기반 요청 추적
- 예약 상태 변경 이력
- 관리자 작업 로그
- 장애 발생 시 확인할 로그 기준 정리

## 전체 구조

```text
Client
  -> Spring Boot API
      -> Auth
      -> Product / Schedule / Seat
      -> Reservation
      -> Payment
      -> Admin Search
      -> Batch
  -> MySQL
  -> Redis
```

## 트랜잭션 경계

### 예약 생성

```text
1. 좌석 조회
2. 좌석 비관적 락 획득
3. 좌석 예약 가능 여부 확인
4. 예약 생성
5. 좌석 임시 점유
6. 예약 상태 이력 저장
7. 트랜잭션 커밋
```

### 결제 승인

```text
1. 결제 승인 요청 수신
2. 결제 결과 검증
3. 결제 정보 저장
4. 예약 상태 CONFIRMED 변경
5. 예약 상태 이력 저장
6. 트랜잭션 커밋
```

### 결제 대기 만료

```text
1. PAYMENT_PENDING 상태 중 만료 시간이 지난 예약 조회
2. 예약 상태 EXPIRED 변경
3. 좌석 점유 해제
4. 예약 상태 이력 저장
5. 트랜잭션 커밋
```

## 동시성 제어

1차 구현에서는 DB 비관적 락을 사용합니다.

선택 이유:

- 예약 도메인은 중복 예약 방지가 성능보다 우선입니다.
- MySQL 트랜잭션과 함께 정합성을 보장하기 쉽습니다.
- 구현과 테스트가 명확해 면접에서 설명하기 좋습니다.

2차 개선에서는 Redis 분산 락과 비교합니다.

## 성능 개선 대상

관리자 예약 검색 API를 성능 개선 대상으로 둡니다.

개선 흐름:

1. 단순 JPA 조회
2. N+1 발생 확인
3. DTO Projection 적용
4. 복합 인덱스 추가
5. 실행계획 비교
6. k6 부하 테스트
7. 성능 개선 보고서 작성

---

## 확장 방향

1차 구현은 예약·결제·만료·검색 성능·동시성 테스트에 집중합니다.

2차 개선에서는 기존 포트폴리오 주제를 더 강하게 연결합니다.

- Redis 분산 락으로 DB 비관적 락과 비교
- 예약/결제 이벤트를 Kafka 또는 SQS로 분리
- 결제 대기 만료 배치를 Chunk 기반으로 개선
- Prometheus/Grafana 또는 CloudWatch로 운영 지표 정리
- 정산 배치와 성능 리포트 추가
