# SeatHub

예약·결제 흐름에서 발생하는 동시성, 상태 전이, 조회 성능, 운영 추적 문제를 다루는 백엔드 프로젝트입니다.

공연, 클래스, 시설 예약 서비스를 가정하고 사용자가 특정 회차의 좌석을 예약한 뒤 결제를 완료하거나, 결제 대기 시간이 지나면 예약이 만료되는 흐름을 구현합니다.

이 프로젝트는 단순 CRUD 구현보다 아래 문제를 직접 다루는 것을 목표로 합니다.

- 같은 좌석에 여러 사용자가 동시에 예약을 요청할 때 중복 예약을 막을 수 있는가
- 예약 생성, 결제 승인, 결제 실패, 예약 만료의 상태 전이를 명확히 관리할 수 있는가
- 외부 결제 시스템 호출과 DB 트랜잭션 경계를 분리할 수 있는가
- 관리자 예약 검색 API에서 N+1, 인덱스, 페이징 문제를 확인하고 개선할 수 있는가
- 장애나 고객 문의가 발생했을 때 예약 상태 변경 이력을 추적할 수 있는가
- 테스트와 문서를 통해 구현 근거를 재현 가능하게 남길 수 있는가

## 기존 프로젝트 경험 재적용

SeatHub는 새로 만든 토이프로젝트 하나로 끝내지 않고, 기존에 진행했던 백엔드 프로젝트에서 다룬 주제를 하나의 실무형 예약·결제 도메인에 다시 적용하는 것을 목표로 합니다.

| 기존 프로젝트 | SeatHub에 반영할 내용 |
| --- | --- |
| One-Core-Architecture | 예약, 결제, 상품, 좌석의 도메인 경계를 나누고 외부 결제 연동 같은 변경 가능성이 큰 부분을 코어 도메인 밖으로 분리 |
| Data-Flux | 결제 대기 만료 배치, 대량 테스트 데이터 생성, 관리자 검색 성능 측정과 개선 리포트 작성 |
| MassFlux-Gateway | 고처리량 요청 상황에서 병목을 확인하고, 예약 생성과 결제 후속 처리를 분리하는 구조 검토 |
| kafka-practice | 예약 생성, 결제 완료, 결제 실패, 예약 만료 이벤트를 비동기 처리로 확장할 수 있는 구조 설계 |
| SimpleIoT.Gateway | 운영 환경에서 요청 흐름과 상태 변화를 추적하는 로그/상태 관리 관점 참고 |

기존 프로젝트의 코드를 그대로 옮기기보다, 각 프로젝트에서 다룬 설계 판단과 검증 방식을 SeatHub 도메인에 맞게 다시 구현합니다.

## 예약 상태 흐름

```text
READY
  -> PAYMENT_PENDING
  -> CONFIRMED
  -> CANCELLED
  -> EXPIRED
  -> PAYMENT_FAILED
```

상태 전이 기준은 다음과 같습니다.

- `READY`: 좌석이 예약 가능한 상태
- `PAYMENT_PENDING`: 예약이 생성되었고 결제 대기 중인 상태
- `CONFIRMED`: 결제가 승인되어 예약이 확정된 상태
- `CANCELLED`: 사용자가 예약을 취소한 상태
- `EXPIRED`: 결제 대기 시간이 지나 자동 만료된 상태
- `PAYMENT_FAILED`: 결제 실패로 예약이 확정되지 않은 상태

## 설계 기준

### 동시성

동일 좌석에 여러 요청이 동시에 들어오는 상황을 가정하고, DB 비관적 락을 사용해 중복 예약을 방지합니다. 이후 Redis 분산 락과 비교해 장단점을 정리할 예정입니다.

### 트랜잭션 경계

외부 결제 API 호출을 DB 트랜잭션 안에서 오래 유지하지 않고, 예약 생성과 결제 승인을 분리합니다. 결제 성공, 실패, 만료는 각각 예약 상태 전이와 상태 이력으로 남깁니다.

### 성능 개선

관리자 예약 검색 API에서 N+1 문제를 재현하고, QueryDSL DTO Projection과 복합 인덱스를 적용해 개선 전후를 비교합니다.

### 운영 추적성

예약 상태 변경 이력과 traceId를 통해 고객 문의나 장애 발생 시 요청 흐름을 추적할 수 있게 만듭니다.
